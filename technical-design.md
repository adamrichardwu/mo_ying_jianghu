# 墨影江湖 - 技术文档

## 1. 文档目标

本文档用于将当前《墨影江湖》的设计稿转化为可执行的技术方案，重点覆盖：

- 项目技术路线
- 动画与美术素材的技术拆分
- 程序系统拆分与编码分析
- 首版 MVP 的实现优先级
- 后续扩展时的技术风险控制

---

## 2. 设计稿技术分析

从当前设计稿来看，项目具备以下技术特征：

### 2.1 低剧情绑定，高系统驱动
这意味着项目首版不需要复杂任务树和剧情编辑器，技术重点应放在：

- 角色成长系统
- 武学系统
- 回合制战斗系统
- 游历事件系统
- 掉落与收集系统

### 2.2 回合制战斗降低了动作系统复杂度
战斗不是实时动作，因此：

- 不需要复杂碰撞判定
- 不需要高频输入响应系统
- 不需要复杂连招输入

但仍然需要：

- 稳定的行动顺序计算
- 明确的技能结算流程
- 状态系统与数值系统
- 技能演出与反馈系统

### 2.3 武学系统是核心，必须数据驱动
由于存在大量可收集武学，且武学分为轻功、内功、外功三类，因此程序层面必须采用：

- 配置驱动
- 模块化结算
- 可扩展的数据结构

如果将武学逻辑硬编码在脚本中，后期会难以维护。

### 2.4 横板场景是表现形式，不应过早做成重动作玩法
当前“2D 横板”更适合作为：

- 场景表现形式
- 地图探索形式
- 战斗演出载体

不建议首版就实现复杂平台跳跃、地形战斗、高精度碰撞等功能。

---

## 3. 技术路线建议

结合当前仓库结构（TypeScript），首版建议技术路线如下：

### 3.1 推荐方案
- **语言：** TypeScript
- **框架 / 引擎：** Phaser 3
- **构建工具：** Vite
- **数据配置：** JSON
- **测试：** Vitest
- **代码规范：** ESLint + Prettier

### 3.2 选择 Phaser 的原因
适合当前项目的原因：

- 对 2D 游戏支持成熟
- 与 TypeScript 结合自然
- 适合做场景切换、UI、战斗演出
- 学习成本较低
- 首版可以快速验证玩法闭环

### 3.3 首版目标平台
建议先做：

- **PC 浏览器版**
- 后续可再适配桌面封装或移动端

这样可以先验证玩法，不必一开始处理复杂平台适配问题。

## 4. 动画技术分析

由于项目是 **2D 横板漫画风 + 回合制战斗**，动画需求应遵循两个原则：

- **优先满足可读性**
- **优先复用，避免首版动画量爆炸**

---

## 5. 角色动画拆分

## 5.1 玩家角色基础动画
首版玩家角色建议最少包含以下通用动画：

### 探索场景动画
- 待机
- 行走
- 跑动（可选）
- 受击
- 倒地 / 失败
- 打坐 / 修炼
- 交互动作

### 战斗场景动画
- 战斗待机
- 普通攻击
- 受击
- 闪避
- 招架
- 释放技能
- 胜利姿态
- 战败姿态

### 技术建议
首版建议采用：
- **逐帧 Sprite Sheet**
- 不建议首版就上骨骼动画系统
- 每个动作控制在 4~10 帧
- 关键动作优先做“起手 - 命中 - 收招”三段结构

---

## 5.2 外功动画需求

外功分流派，因此动画应分为“共用底层动作 + 流派特效”。

### 拳法
需要的演出重点：
- 短促出拳
- 连击节奏
- 命中震动
- 破防反馈

建议动作：
- 直拳
- 连打
- 重击
- 反击姿态

### 剑法
需要的演出重点：
- 横斩
- 竖斩
- 突刺
- 剑气类特效

建议动作：
- 挥剑
- 突刺
- 收势
- 剑意爆发特效

### 刀法
需要的演出重点：
- 大开大合
- 重击停顿
- 压迫感

建议动作：
- 横斩
- 劈砍
- 蓄力斩
- 破甲命中特效

### 枪棍类
需要的演出重点：
- 中距离打击
- 扫击
- 压制感
- 控制效果反馈

建议动作：
- 前刺
- 横扫
- 挑击
- 击退命中特效

### 奇门兵器
首版建议先不做太多独立骨架，采用：
- 通用攻击动作
- 独立武器特效贴图
- 独立命中特效

---

## 5.3 轻功动画需求

轻功不是主要攻击动作，但需要影响战斗节奏与角色表现。

建议轻功演出包括：
- 快速位移
- 残影
- 闪避
- 起势 / 收势
- 空中轻踏感（表现即可，不必真实跳跃系统）

技术建议：
- 轻功效果主要通过 **位移插值 + 残影贴图 + 特效层**
- 不必首版为每个轻功制作完整独立动作集

---

## 5.4 内功动画需求

内功更适合作为状态演出，而不是复杂动作演出。

建议表现：
- 运功站姿
- 真气流动光效
- 护体气罩
- 爆气
- 回气脉冲

技术建议：
- 以内功特效层为主
- 角色本体动作可复用“调息 / 站立 / 施法”
- 使用颜色区分不同内功风格：
  - 刚猛型：金、赤
  - 阴柔型：蓝、紫
  - 中正平和：白、青

---

## 5.5 敌人动画需求

首版敌人不应做过多差异化动作集，建议采用模板化设计。

### 普通敌人模板
- 待机
- 攻击
- 受击
- 倒地

### 精英敌人模板
- 待机
- 普攻
- 技能 1
- 技能 2
- 受击
- 倒地

### 首领模板
- 待机
- 普攻
- 2~3 个核心技能
- 受击
- 特殊蓄力
- 倒地 / 狂暴

---

## 6. 特效素材技术分析

项目的战斗表现高度依赖特效，因此需要单独规划特效素材。

## 6.1 必要特效类型

### 战斗命中特效
- 斩击火花
- 拳掌冲击波
- 破甲裂纹
- 暴击闪光
- 格挡火花

### 状态特效
- 流血
- 内伤
- 护体
- 迟滞
- 封脉
- 凝神

### 资源变化特效
- 真气恢复
- 真气消耗
- 修为增加
- 武学领悟
- 境界突破

### 场景特效
- 风
- 雾
- 落叶
- 火把
- 雨雪（后续）

## 6.2 技术建议
- 特效优先做成 **独立 Sprite Sheet**
- 尽量不要把特效烘焙进角色动画
- 命中特效、状态特效、场景特效分别管理
- 特效层级应独立于角色层，便于复用和开关

## 9.5 战斗模块技术分析

战斗系统是首版最关键的程序部分。

### 建议拆分为以下流程
1. 战斗初始化
2. 生成参战单位
3. 计算回合顺序
4. 选择行动
5. 校验资源与条件
6. 执行技能效果
7. 结算状态
8. 更新 UI
9. 判定胜负
10. 发放奖励

### 需要单独封装的系统
- **TurnOrderSystem**：行动顺序
- **ActionResolver**：技能执行
- **DamageCalculator**：数值计算
- **StatusSystem**：状态生效与回合扣减
- **BattleAI**：敌人行为选择

### 重点要求
- 战斗结算必须是纯逻辑，可脱离画面测试
- 动画播放只负责展示，不参与真实数值判断
- 所有技能效果都应通过统一接口触发

---

## 9.6 武学系统技术分析

这是项目差异化的核心系统。

### 难点
- 三类武学同时装备
- 武学之间有联动
- 武学升级后可能改变效果
- 需要支持大量新增内容

### 技术建议
使用“配置 + 效果管线”方式设计。

即：
- 武学本身是数据
- 武学效果是可组合逻辑
- 结算时读取效果列表逐条执行

### 示例效果类型
- 直接伤害
- 回复真气
- 增加闪避
- 附加状态
- 改变行动顺序
- 触发追击
- 触发护体

这样新增武学时，大多数情况只需要新增配置，不需要新增脚本。

---

## 9.7 修炼系统技术分析

修炼系统本质是“非战斗成长系统”。

### 技术组成
- 修为获取
- 武学熟练度
- 境界条件校验
- 时间推进
- 顿悟随机事件

### 技术建议
首版做成简单数值系统：
- 修炼消耗时间
- 提升熟练度与属性
- 满足条件后解锁下一阶段

不要一开始做复杂日历系统和大量时间轴事件。

---

## 9.8 游历系统技术分析

游历部分建议做成：

- 横板场景移动
- 固定交互点
- 随机遇敌
- 随机事件触发

### 不建议首版做的内容
- 大地图自由寻路
- 复杂 NPC 日程
- 大量动态任务链
- 实时开放世界逻辑

### 建议实现方式
- 每张地图定义可交互节点
- 节点可触发战斗、掉落、文本事件、商店
- 敌人与事件按权重随机抽取

---

## 10. 数值与配置管理建议

## 10.1 数据驱动范围
以下内容应全部配置化：

- 武学
- 敌人
- 物品
- 掉落表
- 事件
- 场景节点
- 商店商品
- 修炼消耗
- 境界门槛

## 10.2 配置文件格式
首版建议用 JSON。

原因：
- 工具链成熟
- TypeScript 读取方便
- 便于后续做表格导出

后续如内容量变大，可转为：
- Excel -> JSON 导出流程

---

## 11. 存档技术分析

存档必须尽早规划，否则后面数据结构容易失控。

### 存档内容
- 角色属性
- 背包
- 已学武学
- 已装备武学
- 武学熟练度
- 图鉴进度
- 当前地图位置
- 已触发事件
- 游戏设置

### 技术建议
- 使用单一 JSON 存档结构
- 预留 `version` 字段
- 读档时做版本兼容处理
- 战斗中尽量避免即时覆盖存档

- 武学学习条件

## 13. 首版 MVP 实现优先级

## 第一阶段：核心玩法验证
目标：做出“可玩闭环”

实现：
- 基础角色数据
- 武学装备系统
- 1 张场景地图
- 3~5 个敌人
- 基础回合制战斗
- 基础掉落
- 基础修炼

## 第二阶段：内容扩展
目标：验证“收集 + 搭配”乐趣

实现：
- 10~20 个武学
- 不同外功流派
- 轻功与内功差异化
- 图鉴与背包
- 事件系统
- 基础商店

## 第三阶段：表现优化
目标：提升完成度

实现：
- 技能特效补齐
- UI 完整化
- 音效与 BGM
- 更多地图
- 精英与首领

---

## 14. 技术风险与控制

## 14.1 风险：动画量过大
控制方式：
- 通用动作复用
- 特效替代独立动作
- 首版不做过多独立武器骨架

## 14.2 风险：武学逻辑失控
控制方式：
- 数据驱动
- 统一效果系统
- 禁止大量技能写死在 UI 或角色脚本中

## 14.3 风险：数值难平衡
控制方式：
- 所有公式集中管理
- 配置表统一导出
- 用测试覆盖关键战斗逻辑

## 14.4 风险：素材管理混乱
控制方式：
- 统一命名
- 统一目录
- 建立资源清单表
- 资源导入时自动校验尺寸和命名

---

## 15. 结论

《墨影江湖》的首版技术重点不是“大世界”或“复杂剧情”，而是：

- 用数据驱动的方式实现武学系统
- 用稳定可测的方式实现回合制战斗
- 用较低成本的动画与特效完成漫画风表现
- 用模块化结构支持后续内容扩展

从技术上看，最优先实现顺序应为：

1. 数据结构
2. 战斗逻辑
3. 武学与修炼
4. 探索与事件
5. 存档
6. 表现优化

只有先把这条主线打通，后续素材扩展和玩法扩展才不会返工。
